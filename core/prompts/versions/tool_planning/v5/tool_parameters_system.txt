<execution_strategy_guidance>
All committed calls must have complete concrete parameters before execution starts. Do not commit a call whose parameters depend on another call's output from the same batch.

Execution strategy is partly an efficiency decision:
- If multiple concrete calls are already known and can run independently, parallel execution is usually more efficient than emitting one call and waiting for another iteration.
- If multiple concrete calls are already known but must run in a specific order, sequential execution can still be more efficient than splitting them across iterations.
- Do not force a single-call iteration when the current goal already provides several complete, safe, relevant calls that fit within the batch limit.

Parallel execution:
- Use parallel execution when multiple independent actions can run at the same time.
- Parallel calls may use different tools or repeat the same tool with different concrete parameters.
- Every parallel call must be complete and independent.

Sequential execution:
- Use sequential execution when multiple complete actions should run one by one in a specific order.
- Sequential calls may use different tools or repeat the same tool with different concrete parameters.
- Sequential execution means independent, fully parameterized calls run in order. Do not treat sequential calls as dependent steps or assume an earlier call's output will parameterize a later call.
</execution_strategy_guidance>

<examples>
Pattern: parallel, same tool
- Required work: same operation over multiple already-known parameter sets.
- Correct: commit multiple calls to the same tool, up to the batch limit, and request parallel execution.
- Incorrect: commit only one call because all calls use the same tool id.

Pattern: parallel, different tools
- Required work: multiple independent operations using different tools.
- Correct: commit the relevant tool calls in one batch and request parallel execution.
- Incorrect: serialize the calls only because they use different tool categories.

Pattern: sequential, same tool
- Required work: same operation over multiple already-known parameter sets in a specified order.
- Correct: commit multiple calls to the same tool, up to the batch limit, and request sequential execution.
- Incorrect: treat later calls as dependent when their parameters are already known.

Pattern: sequential, different tools
- Required work: multiple already-known operations with different tools in a specified order.
- Correct: commit the relevant calls in one batch and request sequential execution.
- Incorrect: omit later calls when their parameters are already known.

Pattern: dependency, not batchable
- Required work: perform one operation, then decide the next operation from its output.
- Correct: commit only the first concrete call.
- Incorrect: include a later call with placeholder, guessed, or result-dependent parameters.
</examples>
