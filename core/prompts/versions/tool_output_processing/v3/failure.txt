Interpretation mode guide:
- command_output: interpret as command/tool result output.
- metadata_only: interpret as metadata output (for example stat/path metadata), not file contents.
- file_contents: interpret as file/content read output.

The output below contains errors. Your job is to:
1. Extract the root cause of the failure from stderr/error messages.
2. Identify what parameters or conditions caused the failure.
3. Preserve explicit relationships such as parameter -> error, path -> error, or tool state -> failure.
4. Report only what is explicitly present in the tool output.
5. Preserve every distinct warning/error/failure/status line that changes interpretation or next action.
6. Aggregate repeated errors only when they are exact repeats or clearly identical messages.

Required output format (respond with ONLY valid JSON):
{{
  "summary": "Brief factual summary of the failure from the output",
  "key_findings": ["error_message_1", "error_message_2", "root_cause_or_conflict"],
  "structured_signals": [

  ],
  "decision_evidence": ["short raw excerpt 1"],
  "lossiness_risk": "medium"
}}

Field rules:
- `summary`: factual failure summary only
- `key_findings`: distinct failure causes, parameter conflicts, and explicit failure conditions
- `structured_signals`: use only these types when supported by evidence:
  - `error_context`
  - `kv_pair`
  - `path`
  - `redirect`
  - `endpoint`
- `decision_evidence`: 1-5 short raw excerpts only when they directly support the failure diagnosis
  - If the raw output includes evidence locators such as `path:line:excerpt`, `line:excerpt`, `line| excerpt`, grep-style matches, numbered file excerpts, or citation-like references, preserve the locator together with the excerpt.
  - Do not strip file paths, line numbers, or citation prefixes from evidence excerpts when they are present.
- `lossiness_risk`: `low`, `medium`, or `high`

Constraints:
- Extract all distinct error causes from stderr
- Do not collapse multiple distinct warnings/errors into one generic finding; include each distinct message in `key_findings`, `decision_evidence`, or `structured_signals` as appropriate
- If an error repeats many times, summarize it with count/frequency instead of repeating lines
- Be specific about parameter conflicts when explicit in the output
- If sampling mode is `head_middle_tail_sample`, treat input as non-contiguous excerpts and avoid claiming the list of errors is exhaustive unless the text explicitly states totals
- Do NOT infer vulnerabilities, impact, remediation steps, or next actions
- Do NOT add findings not explicitly present in the output
- Do not include any explanatory text before or after the JSON
- If a field has no content, return an empty array instead of omitting it
