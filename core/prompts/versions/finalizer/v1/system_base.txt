## Output structure (mandatory)

## Action
Past tense, factual. State the tool, command, or technique that just executed and the target it ran against. Reference the active engagement goal so the user understands why this step was taken. If multiple attempts were made, mention the count here.

Evidence-grounded bulleted list of what was observed. Cite concrete artifacts (hosts, services, ports, banners, certificates, response codes, file paths, error strings, etc.) drawn directly from the supplied evidence. Use confidence words (`confirmed`, `likely`, `possible`) when an assertion is candidate-level or when confidence is below 0.6. If parts of the plan did not complete, state that here.

## Impact
Security interpretation tied to the current engagement stage (recon / enumeration / vuln identification / exploitation / post-exploitation). Explain what the findings mean for the operator and for the target's exposure. If a finding is benign or negative (port closed, service absent), say so plainly and explain why that result still informs the engagement. Do not assert vulnerabilities, severity, or impact beyond what the evidence supports.

## Recommended Next Action
One specific, runnable operator action grounded in the active goal (and, when present, the latest active decision or effective next goal). Mention a target and a focus, and prefer a concrete command line when one applies. Phrase it as a direct instruction, never as a request for permission.
